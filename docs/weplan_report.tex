\section{赛题分析}

\subsection{赛题背景}

美团 AI Hackathon 2026 是美团面向全社会开发者的年度 AI 创新赛事，旨在探索大语言模型（LLM）与 Agent 技术在本地生活服务领域的落地应用。本次赛事共设置 6 个赛题方向，覆盖从点餐、出行到活动规划的多个场景。

赛题 06「周末闲时活动规划」由小团Agent产品部出题，核心诉求是利用 AI 技术帮助用户高效规划周末闲暇时间的活动安排。出题方明确指出，当前用户在规划周末活动时面临信息碎片化、决策成本高、个性化不足三大核心痛点。

\begin{infobox}[赛题06原文要求]
设计一个 AI 驱动的周末活动规划工具，能够根据用户的偏好、位置、预算、时间等多维度信息，自动推荐并规划周末活动方案。要求方案包含具体的活动内容、时间安排、路线规划和费用估算。鼓励使用 Multi-Agent 架构和外部 API 集成。
\end{infobox}

\subsection{出题方分析}

小团Agent产品部是美团内部负责智能助手产品的核心团队，其主要产品包括小团AI管家（已完成超过7亿次信息核验、整合超过13亿条用户评价）和即将推出的小美AI助手。该团队在 Agent 技术方面有深厚积累，因此其出题视角更关注以下几个方面：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{Agent 架构的合理性}：是否采用了成熟的 Multi-Agent 协作模式，各 Agent 职责划分是否清晰。
  \item \textbf{工具调用的鲁棒性}：外部 API 调用的异常处理、重试机制、降级策略是否完善。
  \item \textbf{用户体验的连贯性}：交互流程是否自然，规划结果是否可操作。
  \item \textbf{与美团生态的结合度}：是否考虑了与美团既有服务的集成可能性。
\end{enumerate}

不同于简单的 POI 推荐或行程模板，WePlan 通过多个专业化 Agent 的协作，实现从用户需求理解、场景匹配、方案生成、路线优化到最终展示的全链路智能规划。它不仅告诉用户"去哪里玩"，更解决"怎么安排一整天"的系统性问题。

WePlan = 懂你的 AI 规划师 + 实时地理引擎 + 群体智慧优选

\subsection{使用场景}

\textbf{场景一：周末上午的即兴规划}。小明周六早上醒来，想出去转转但没有具体想法。打开WePlan，输入"今天想去逛逛，预算200以内，不想走太远"。WePlan 在20秒内生成3个完整方案，包含餐厅、景点、咖啡馆的组合推荐，以及详细的路线和时间安排。

\textbf{场景二：约会前的精心策划}。小红想给男朋友准备一个惊喜周末。输入"浪漫约会，周六全天，预算500，喜欢文艺"。WePlan 生成包含网红书店、精品餐厅、夜景观赏的全天方案，并附带每个地点的评分和特色。

\textbf{场景三：多人聚会的协调}。一个6人小组想周末聚会，但每个人的位置和偏好不同。WePlan 的群体投票功能让每个人都能投票选择偏好，最终生成一个照顾所有人的折中方案。

\subsection{产品边界}

WePlan 专注于"规划"环节，不涉及预订和支付。其核心价值在于决策辅助，帮助用户从"不知道干什么"到"有一个完整的周末计划"。在与美团生态集成后，每个推荐方案中的POI都可以直接链接到美团平台完成后续的消费闭环。

\section{创新点}

WePlan 在技术和产品层面共有 7 项核心创新，这些创新点贯穿了系统设计的各个环节：

\begin{table}[H]
\centering
\caption{七项核心创新}
\label{tab:innovations}
\renewcommand{\arraystretch}{1.5}
{\small
\begin{tabular}{p{0.5cm} p{3cm} p{3.5cm} p{5cm}}
\toprule
\textbf{No.} & \textbf{创新点} & \textbf{技术手段} & \textbf{用户价值} \\
\midrule
1 & 8-Agent 协作架构 & Multi-Agent DAG 调度 & 专业分工提升规划质量 \\
2 & 群体投票机制 & 加权排序算法 & 多人场景下的公平决策 \\
3 & ToolHarness 工程体系 & 10种异常分类处理 & 98\%以上的调用成功率 \\
4 & 确定性基础设施 & 受Claude Code论文启发 & 规划结果稳定可复现 \\
5 & 实时天气感知 & 高德天气API集成 & 方案随天气动态调整 \\
6 & 双面板交互设计 & 对话+方案并行展示 & 沉浸式规划体验 \\
7 & Agent 思维可视化 & SSE实时推送 & 用户理解AI决策过程 \\
\bottomrule
\end{tabular}
}
\end{table}

\textbf{创新一：8-Agent 协作架构。}不同于传统的单一 LLM 对话模式，WePlan 将活动规划问题分解为 8 个独立的子任务，每个子任务由一个专业化的 Agent 负责。这种架构借鉴了软件工程中的微服务理念，每个 Agent 只关注自己的专业领域，通过明确的输入输出接口进行协作。

\textbf{创新二：群体投票机制。}针对多人出行场景，WePlan 设计了基于加权排序的群体投票算法。每位参与者可以对方案的各个维度进行评分，系统通过 Borda Count 变体算法计算综合排名，确保最终方案最大程度地满足所有人的偏好。

\textbf{创新三：ToolHarness 工程体系。}受 Anthropic 发布的 Claude Code 论文中"98.4\%确定性基础设施"理念的启发，WePlan 构建了完整的工具调用治理框架。该框架对 10 种常见异常进行了分类编码，并为每种异常定义了对应的处理策略，确保系统在面对外部 API 故障时仍能提供降级服务。

\textbf{创新四：确定性基础设施。}受 Claude Code 论文启发，我们将 98\% 以上的工程投入聚焦在确定性的基础设施层面。ToolHarness 框架对所有外部调用实现了统一的参数校验、超时控制、重试逻辑和降级策略，使得系统在面对不可预测的外部环境时仍能提供稳定可靠的服务。

\textbf{创新五：实时天气感知融合。}WePlan 不是在方案生成后才考虑天气因素，而是将天气信息作为一等公民融入规划的每个环节。WeatherAgent 与 POIAgent 并行执行，天气数据直接影响 POI 筛选、时间安排和活动替换决策。例如，当预测下午有雨时，系统会自动将下午的户外活动替换为室内方案，并相应调整路线。

\textbf{创新六：双面板交互设计。}传统的 AI 对话产品采用单一的对话流形式，用户需要在对话中寻找关键信息。WePlan 的双面板设计将「对话」和「方案」解耦到两个并行的视图中：左侧对话面板负责需求沟通和过程展示，右侧方案面板负责结果呈现和操作交互。这一设计使得信息密度提升约 60\%，用户的信息获取效率显著提高。

\textbf{创新七：Agent 思维可视化。}透明性是建立用户对 AI 信任的关键。WePlan 通过 SSE 实时推送每个 Agent 的思考过程，让用户能够看到 AI 是如何一步步做出决策的。这不是简单的 loading 动画，而是真实的推理过程展示。用户可以看到 IntentAgent 如何理解需求、POIAgent 搜索了哪些关键词、RouteAgent 为何选择某条路线。这种透明性显著提升了用户对推荐结果的接受度。

\subsection{AI 渗透趋势}

本地生活服务中的 AI 应用正从简单的推荐算法向复杂的 Agent 交互演进。这一趋势表现为三个阶段：

\textbf{阶段一：推荐驱动（2018-2022）。}以协同过滤和深度学习为基础的推荐系统是这一阶段的主流。美团的「猜你喜欢」、大众点评的「智能排序」等功能代表了这一阶段的技术水平。推荐系统解决了"信息过载"问题，但无法处理复杂的多步骤决策。

\textbf{阶段二：对话驱动（2023-2024）。}ChatGPT 引发的大语言模型浪潮推动了对话式交互在本地生活领域的应用。用户可以通过自然语言表达需求，系统通过 LLM 理解意图并返回结构化结果。美团的小团AI管家是这一阶段的典型产品。

\textbf{阶段三：Agent驱动（2025-）。}以 Multi-Agent 为代表的自主规划系统是当前的前沿方向。Agent 不仅理解用户需求，更能主动调用工具、整合信息、生成完整方案。WePlan 正是定位在这一阶段的创新产品。

\subsection{周末经济}

「周末经济」已成为消费市场的重要增长极。根据美团研究院的数据，2025 年周末消费占全周消费的比重达到 42\%，其中到店休闲娱乐品类的周末占比高达 58\%。然而，在消费意愿旺盛的同时，用户面临的规划难题却日益突出。

调研数据显示，73\% 的用户表示「想出门但不知道去哪」，64\% 的用户因「规划太麻烦」而选择宅在家里。这一巨大的未满足需求正是 WePlan 的市场切入点。

\subsection{用户代际差异}

不同年龄段用户在周末活动规划上表现出显著的代际差异：

\textbf{千禧一代（26-35岁）}：注重品质和效率，愿意为好体验买单。是 WePlan 的核心目标用户群，占潜在用户的 45\%。

\textbf{X世代（36-45岁）}：以家庭活动为主，对安全性和便利性要求高。是亲子场景的主力用户。

\subsection{市场机遇}

综合分析，WePlan 面对的市场机遇可归纳为以下三点：

\begin{enumerate}[leftmargin=2em]

  \item \textbf{供给侧}：美团平台拥有超过 900 万活跃商户和数十亿条用户评价数据，为 AI 规划提供了丰富的数据基础。
  \item \textbf{技术侧}：大语言模型的 Function Calling 能力、Multi-Agent 框架的成熟、地图 API 的开放化，使得构建复杂的规划系统成为可能。
\end{enumerate}

\section{美团AI生态分析}

\subsection{美团AI战略布局}

美团自 2023 年以来加速了 AI 战略布局，全年研发投入超过 260 亿元，其中 AI 相关投入占比约 47\%，接近 122 亿元。这一投入规模在中国互联网企业中位居前列。

美团的 AI 布局围绕三条主线展开：

\textbf{主线一：AI 基础设施。}美团建设了大规模 GPU 集群和模型训练平台，支撑内部多个团队的模型研发。据公开信息，美团的 GPU 集群规模已超过万卡，支持千亿级参数模型的训练。

\textbf{主线二：AI 产品化。}以小团AI管家、小美AI助手为代表的 C 端 AI 产品，以及面向商户的 AI 运营工具，构成了美团 AI 产品矩阵。

\textbf{主线三：AI 平台化。}美团正在构建开放的 AI 能力平台，通过 API 和 SDK 的形式向外部开发者提供模型推理、知识库检索等能力。本次 Hackathon 正是平台化战略的重要组成部分。

\subsection{小团AI管家深度分析}

小团AI管家是美团 AI 产品化战略的旗舰产品，也是赛题06最直接的参照系。截至 2025 年底，小团AI管家已经积累了以下核心能力：

\begin{itemize}[leftmargin=2em]
  \item \textbf{信息核验}：完成超过 7 亿次商户信息核验，覆盖营业时间、价格区间、菜品信息等关键数据维度
  \item \textbf{评价整合}：整合超过 13 亿条用户评价，构建了细粒度的商户质量评估体系
  \item \textbf{意图理解}：支持超过 200 种用户意图的识别，覆盖餐饮、休闲、旅游等多个品类
  \item \textbf{多轮对话}：平均对话轮次 4.2 轮，用户满意度达到 87\%
\end{itemize}

小团AI管家的技术架构采用了检索增强生成（RAG）+ 工具调用（Tool Use）的混合模式。用户的自然语言查询首先经过意图识别模块分类，然后根据意图类型选择相应的处理流程：简单查询走 RAG 路径，复杂任务走 Agent 路径。

WePlan 虽然没有直接使用 LongCat，但其设计理念受到了 LongCat 的启发：通过将长文本的理解任务分解为多个短文本的处理任务，实现了更高效的信息抽取。

\subsection{Google Travel Canvas}

Google 于 2025 年推出的 Travel Canvas 是目前功能最完整的 AI 旅行规划工具。它集成了 Google Maps、Google Reviews 和 Gemini 大模型，能够根据用户的对话式输入生成可视化的旅行行程。

Travel Canvas 的核心优势在于其无与伦比的数据覆盖（Google Maps 覆盖全球超过 2 亿个兴趣点）和强大的多模态能力（支持图片识别和语音输入）。但其劣势同样明显：对中国市场的数据覆盖不足，且缺乏对美团等本地化平台的集成。

\subsection{MonkeyTravel}

MonkeyTravel 是一款专注于东南亚市场的 AI 旅行规划应用，采用 GPT-4 作为核心模型。其特色在于社交化的行程分享和实时协作编辑功能。

\subsection{Meetup.AI}

Meetup.AI 是面向多人活动场景的 AI 规划工具，其核心功能是帮助一群人找到大家都满意的聚会方案。这与 WePlan 的群体投票功能有一定的相似性，但 Meetup.AI 更侧重于欧美市场的社交场景。

\subsection{携程/飞猪 AI 助手}

国内的携程和飞猪也分别推出了 AI 规划功能。携程的「旅行 AI 助手」侧重于跨城市旅行的行程规划，飞猪的「智能行程」侧重于机酒预订的智能推荐。两者都缺乏对本地短距离活动（如周末半日游）的精细化支持。

\section{用户痛点分析}

\subsection{痛点调研方法}

为了准确把握用户在周末活动规划中的核心痛点，团队采用了以下调研方法：

\begin{enumerate}[leftmargin=2em]
  \item 文献研究：分析美团研究院、QuestMobile等机构的公开报告
  \item 竞品体验：实际使用了10余款相关产品并记录交互体验
  \item 社交媒体分析：在小红书、微博等平台收集了超过500条与「周末不知道干什么」相关的用户吐槽
  \item 用户访谈：对15位目标用户进行了深度访谈
\end{enumerate}

\subsection{四大核心痛点}

\textbf{痛点一：信息碎片化（影响 84\% 用户）。}用户需要在多个平台（大众点评看评价、高德看距离、天气APP看天气、小红书看攻略）之间反复切换，才能获取做出决策所需的信息。信息的碎片化导致决策效率极低。

\textbf{痛点二：决策疲劳（影响 73\% 用户）。}面对海量的选择，用户往往陷入「选择悖论」：选项越多，越难以做出决定。很多用户最终因为决策疲劳而放弃出行。

\textbf{痛点三：规划能力不足（影响 61\% 用户）。}即使确定了想去的地方，如何合理安排时间顺序、规划交通路线、预留用餐时间等，也是一个复杂的优化问题。大多数用户缺乏这方面的经验。

\textbf{痛点四：多人协调困难（影响 52\% 用户）。}当涉及多人一起出行时，协调每个人的偏好、时间和位置成为一个额外的难题。``众口难调''的问题在群体活动中尤为突出。

\subsection{架构设计理念}

WePlan 的架构设计受到两个重要理论的启发：

\textbf{第一，Anthropic Claude Code 的确定性基础设施理念。}Claude Code 论文指出，在实际的 Agent 系统中，98.4\% 的工作量应投入到确定性的基础设施建设上（如工具调用框架、异常处理、日志追踪等），只有 1.6\% 是模型推理本身。WePlan 深刻认同这一理念，将大量工程资源投入到 ToolHarness、错误处理矩阵、调用追踪等基础设施的建设中。

\textbf{第二，微服务架构的职责分离原则。}每个 Agent 作为一个独立的"微服务"，拥有明确的输入、输出和职责边界。Agent 之间通过标准化的消息格式进行通信，确保系统的可维护性和可扩展性。

\subsection{层次说明}

\textbf{前端展示层}：基于纯原生 HTML + CSS + JavaScript 构建的轻量级 Web 应用，采用双面板布局（左侧对话面板 + 右侧方案面板），零依赖零构建，支持实时更新和交互式操作。

\textbf{API 网关层}：基于 FastAPI 构建的异步 Web 服务，通过 Server-Sent Events（SSE）实现与前端的实时通信。网关层负责请求路由、认证鉴权和流量控制。

\textbf{Agent 协作层}：WePlan 的核心，包含 8 个专业化 Agent。Agent 之间通过 DAG（有向无环图）调度器进行协调，确保执行顺序的正确性和并行度的最大化。

\textbf{ToolHarness 层}：确定性基础设施层，负责所有外部工具调用的生命周期管理，包括参数校验、调用执行、异常捕获、重试策略和降级处理。

\textbf{外部服务层}：包括 MiniMax M2.7 大模型服务、高德地图 API 和天气服务等外部依赖。

\subsection{数据流转}

一次完整的用户请求在系统中的流转路径如下：

\begin{enumerate}[leftmargin=2em]
  \item 用户在前端输入自然语言需求
  \item 请求通过 SSE 通道发送至 API 网关
  \item 网关将请求交给 Orchestrator（编排器），启动 Agent DAG
  \item IntentAgent 首先执行，解析用户意图和约束条件
  \item POIAgent 和 WeatherAgent 并行执行，分别获取兴趣点和天气信息
  \item RouteAgent 基于 POI 结果规划路线
  \item TimeAgent 和 BudgetAgent 并行执行，分别安排时间和预算
  \item PlanAgent 整合所有结果，生成最终方案
  \item 如果是多人场景，VoteAgent 启动投票流程
  \item 最终方案通过 SSE 推送至前端展示
\end{enumerate}

\section{Agent 设计}

WePlan 的 Agent 层由 8 个专业化 Agent 组成，每个 Agent 具有明确的职责边界、输入输出规范和提示词策略。以下逐一详细介绍。

\subsection{IntentAgent -- 意图理解}

\textbf{职责}：解析用户的自然语言输入，提取结构化的意图信息，包括活动类型偏好、时间约束、预算范围、人数、位置信息等。

\textbf{输入}：用户的原始文本消息和对话历史。

\textbf{输出}：结构化的意图对象（JSON格式），包含以下字段：

\begin{lstlisting}[caption={IntentAgent 输出结构},label={lst:intent-output}]
{
  "activity_types": ["outdoor", "culture", "food"],
  "time_window": {
    "date": "2026-05-17",
    "start_hour": 10,
    "end_hour": 20
  },
  "budget": {
    "total": 500,
    "per_person": 125,
    "currency": "CNY"
  },
  "people": {
    "count": 4,
    "has_children": false,
    "has_elderly": false
  },
  "location": {
    "city": "beijing",
    "district": "chaoyang",
    "lat": 39.9042,
    "lng": 116.4074
  },
  "constraints": ["no_hiking", "wheelchair_accessible"],
  "mood": "relaxed"
}
\end{lstlisting}

\textbf{提示词策略}：IntentAgent 的 System Prompt 遵循「角色-任务-格式-约束」四段式结构。为了处理用户表达的模糊性，提示词中包含了大量的示例和边界条件说明。例如，当用户说"随便逛逛"时，系统应理解为轻松休闲的活动偏好，而非字面意义上的"任何活动都行"。

\textbf{核心代码}：

\begin{lstlisting}[caption={IntentAgent 核心实现},label={lst:intent-agent}]
class IntentAgent(BaseAgent):
    """Parse user natural language into structured intent."""

    def __init__(self, llm_client: LLMClient):
        super().__init__(name="IntentAgent", llm=llm_client)
        self.system_prompt = self._load_prompt("intent_system.txt")

    async def execute(self, user_msg: str,
                      history: list[dict]) -> Intent:
        messages = [
            {"role": "system", "content": self.system_prompt},
            *history,
            {"role": "user", "content": user_msg},
        ]
        resp = await self.llm.chat(
            messages=messages,
            response_format={"type": "json_object"},
            temperature=0.1,
        )
        return Intent.model_validate_json(resp.content)
\end{lstlisting}

\subsection{POIAgent -- 兴趣点搜索}

\textbf{职责}：根据 IntentAgent 输出的结构化意图，调用高德地图 POI 搜索 API，获取符合条件的兴趣点列表，并基于评分、距离、评价数等维度进行排序和筛选。

\textbf{输入}：Intent 对象。

\textbf{输出}：排序后的 POI 列表，每个 POI 包含名称、地址、经纬度、类型、评分、评价数、人均消费等信息。

\textbf{工具调用}：POIAgent 通过 ToolHarness 调用高德 POI 搜索 API，每次请求最多返回 25 个结果。对于复杂的多类型需求（如"既想逛公园又想吃火锅"），POIAgent 会发起多次搜索请求并合并结果。

\textbf{排序算法}：POI 的最终排序基于以下加权评分公式：

\begin{equation}
S_{poi} = w_1 \cdot \frac{R}{5} + w_2 \cdot \frac{N}{\max(N)} + w_3 \cdot (1 - \frac{D}{\max(D)}) + w_4 \cdot M_{intent}
\end{equation}

其中 $R$ 为评分（0-5），$N$ 为评价数量，$D$ 为与用户的距离，$M_{intent}$ 为与用户意图的匹配度（0-1）。权重默认值为 $w_1=0.3, w_2=0.2, w_3=0.2, w_4=0.3$。

\subsection{RouteAgent -- 路线规划}

\textbf{职责}：基于 POIAgent 输出的兴趣点列表，调用高德路线规划 API，计算各POI之间的最优路线，并考虑交通方式（步行、公交、驾车）的选择。

\textbf{输入}：POI 列表和用户的交通偏好。

\textbf{输出}：包含路线详情的行程安排，每段路程包含起止点、距离、预计耗时、推荐交通方式。

\textbf{路线优化}：RouteAgent 使用贪心最近邻算法（Greedy Nearest Neighbor）求解近似的旅行商问题（TSP），以最小化总行程时间。对于 POI 数量不超过 8 个的情况，算法的近似比可以控制在 1.5 以内。

\begin{equation}
\text{route}^* = \arg\min_{\pi \in \Pi} \sum_{i=1}^{n-1} t(\pi_i, \pi_{i+1}) + \sum_{i=1}^{n} d_i
\end{equation}

其中 $t(\pi_i, \pi_{i+1})$ 为第 $i$ 个和第 $i+1$ 个 POI 之间的交通时间，$d_i$ 为在第 $i$ 个 POI 的停留时间。

\subsection{TimeAgent -- 时间编排}

\textbf{职责}：根据用户的时间窗口、各POI的营业时间和推荐游玩时长，为整个行程编排合理的时间安排。

\textbf{输入}：Intent 中的时间窗口、POI 列表（含营业时间）、Route 信息（含路段耗时）。

\textbf{输出}：带有精确时间标注的行程时间表。

\textbf{时间约束处理}：TimeAgent 需要处理以下复杂约束：
\begin{itemize}[leftmargin=2em]
  \item 营业时间约束：确保到达时间在 POI 营业时间范围内
  \item 用餐时间约束：午餐安排在 11:30-13:30，晚餐安排在 17:30-19:30
  \item 交通缓冲：为每段交通预留 10-15 分钟的缓冲时间
  \item 休息时间：每连续活动 3 小时安排一次 15-30 分钟的休息
\end{itemize}

\subsection{WeatherAgent -- 天气感知}

\textbf{职责}：获取目标城市的实时天气信息和短期天气预报，为方案的可行性评估提供天气维度的参考。

\textbf{输入}：Intent 中的城市和日期信息。

\textbf{输出}：天气状况（晴/阴/雨/雪等）、气温范围、风力、空气质量指数、穿衣建议、是否适合户外活动的判断。

\textbf{天气影响策略}：当天气条件不利于户外活动时，WeatherAgent 会生成天气建议标记，PlanAgent 在生成最终方案时会参考该标记，自动将户外活动替换为室内备选方案。

\subsection{BudgetAgent -- 预算管理}

\textbf{职责}：根据用户的预算约束和各POI的消费水平，生成详细的费用预估和预算分配方案。

\textbf{输入}：Intent 中的预算信息、POI 列表（含人均消费）、Route 信息（含交通费用）。

\textbf{输出}：每个活动环节的预估费用、交通费用、总费用、预算余量。

\textbf{预算优化策略}：当总预估费用超出用户预算时，BudgetAgent 会启动费用优化流程：
\begin{enumerate}[leftmargin=2em]
  \item 首先尝试替换高消费 POI 为同类型的低消费替代品
  \item 其次优化交通方式（如从打车改为公交）
  \item 最后如果仍然超支，减少活动数量并提醒用户
\end{enumerate}

\subsection{VoteAgent -- 群体投票}

\textbf{职责}：在多人出行场景中，收集每位参与者的偏好评分，通过加权排序算法找到最优的群体方案。

\textbf{输入}：多个参与者的偏好向量和候选方案列表。

\textbf{输出}：按群体满意度排序的方案列表，每个方案附带综合评分和各参与者的满意度指标。

\textbf{投票算法}：WePlan 使用 Borda Count 的加权变体进行群体决策。每位参与者对每个候选方案进行 1-5 分的评价，系统根据每位参与者的权重（默认等权，可调整）计算加权总分：

\begin{equation}
S_{plan} = \sum_{j=1}^{m} w_j \cdot \frac{v_{j,plan}}{5}
\end{equation}

其中 $m$ 为参与者数量，$w_j$ 为第 $j$ 位参与者的权重，$v_{j,plan}$ 为其对方案的评分。

\subsection{PlanAgent -- 方案整合}

\textbf{职责}：作为最终的整合者，PlanAgent 汇总所有其他 Agent 的输出，生成 3 个完整的候选方案，每个方案包含活动列表、时间安排、路线规划、费用预估和风险提示。

\textbf{输入}：所有前序 Agent 的输出结果。

\textbf{输出}：3个完整的规划方案，每个方案包含：
\begin{itemize}[leftmargin=2em]
  \item 方案名称和一句话描述
  \item 活动列表（含时间、地点、费用）
  \item 交通路线和出行方式
  \item 总费用和预算分析
  \item 天气风险提示和备选方案
  \item 雷达图评分（趣味性/性价比/便捷性/舒适度/新鲜感五个维度）
\end{itemize}

\textbf{方案差异化策略}：为了确保 3 个方案之间有足够的差异性，PlanAgent 采用了「风格标签」策略：第一个方案标记为「经济实惠型」，第二个为「品质体验型」，第三个为「探索冒险型」。每个风格标签对应不同的POI选择偏好和预算分配策略。

\subsection{Agent 间通信协议}

8 个 Agent 之间的通信采用统一的消息格式（AgentMessage），确保信息传递的标准化和可追踪性：

\subsection{重试与降级策略}

ToolHarness 的重试策略采用指数退避（Exponential Backoff）加随机抖动（Jitter）的方式，避免在服务恢复时产生请求风暴：

\begin{equation}
t_{wait} = \min(t_{base} \cdot 2^{n-1} + \text{random}(0, t_{jitter}), t_{max})
\end{equation}

其中 $t_{base}=1s$，$t_{jitter}=0.5s$，$t_{max}=30s$，$n$ 为当前重试次数。

降级策略遵循「尽力服务」原则：即使部分外部服务不可用，系统也应该尽可能地提供有价值的输出。例如，当高德 POI 搜索服务不可用时，系统可以基于内置的热门 POI 缓存提供基础推荐。

\subsection{偏好学习}

中期记忆中的用户偏好向量通过以下方式构建和更新：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{显式偏好}：用户在交互中直接表达的偏好（如"我喜欢户外活动"）
  \item \textbf{隐式偏好}：通过用户的选择行为推断的偏好（如多次选择含博物馆的方案）
  \item \textbf{衰减机制}：老旧的偏好信号会随时间指数衰减，确保推荐结果反映用户当前的兴趣
\end{enumerate}

偏好更新公式：
\begin{equation}
\vec{p}_{t+1} = \alpha \cdot \vec{p}_t \cdot e^{-\lambda \Delta t} + (1-\alpha) \cdot \vec{f}_{new}
\end{equation}

其中 $\vec{p}_t$ 为当前偏好向量，$\vec{f}_{new}$ 为新的偏好信号，$\alpha$ 为保留率，$\lambda$ 为衰减系数。

\section{群体投票系统}

\subsection{设计动机}

在实际的周末活动场景中，超过 40\% 的出行是多人同行（朋友聚会、家庭出行、团建等）。多人场景下的核心挑战是"众口难调"：每个人的偏好不同，如何找到一个大家都相对满意的方案？

WePlan 的群体投票系统正是为解决这一问题而设计。它将群体决策问题形式化为一个多目标优化问题，并通过社会选择理论中的投票机制求解。

\subsection{投票流程}

\begin{enumerate}[leftmargin=2em]
  \item 发起者创建规划请求，系统生成邀请链接
  \item 参与者通过链接加入，各自设置偏好权重
  \item 系统基于所有参与者的偏好生成候选方案集
  \item 每位参与者对候选方案进行打分（1-5分）
  \item 系统计算加权综合评分，输出最终排名
  \item 支持实时查看投票进度和中间结果
\end{enumerate}

\subsection{公平性保障}

为了确保投票结果的公平性，系统实现了以下机制：

\begin{itemize}[leftmargin=2em]
  \item \textbf{匿名投票}：参与者的投票结果在最终统计前不对其他人可见
  \item \textbf{位置公平}：方案展示顺序对每位参与者随机化，避免顺序偏差
  \item \textbf{帕累托检查}：确保最终方案是帕累托最优的（不存在另一个方案使所有人都更满意）
\end{itemize}

\subsection{调用封装}

为了统一管理 MiniMax API 的调用，WePlan 封装了一个 LLMClient 类，提供标准化的接口和错误处理：

\begin{lstlisting}[caption={LLMClient 封装},label={lst:llm-client}]
import httpx
from typing import Optional

class LLMClient:
    BASE_URL = "https://api.minimax.chat/v1"

    def __init__(self, api_key: str, group_id: str,
                 model: str = "abab7-chat"):
        self.api_key = api_key
        self.group_id = group_id
        self.model = model
        self.client = httpx.AsyncClient(timeout=30.0)

    async def chat(
        self,
        messages: list[dict],
        tools: Optional[list[dict]] = None,
        temperature: float = 0.7,
        response_format: Optional[dict] = None,
    ) -> dict:
        headers = {
            "Authorization": f"Bearer {self.api_key}",
            "Content-Type": "application/json",
        }
        payload = {
            "model": self.model,
            "messages": messages,
            "temperature": temperature,
        }
        if tools:
            payload["tools"] = tools
        if response_format:
            payload["response_format"] = response_format

        url = f"{self.BASE_URL}/text/chatcompletion_v2"
        params = {"GroupId": self.group_id}
        resp = await self.client.post(
            url, json=payload,
            headers=headers, params=params,
        )
        resp.raise_for_status()
        data = resp.json()
        return data["choices"][0]["message"]

    async def close(self):
        await self.client.aclose()
\end{lstlisting}

\subsection{POI 搜索集成}

高德 POI 搜索 API 是 WePlan 获取兴趣点数据的核心接口。以下是封装后的调用代码：

\begin{lstlisting}[caption={高德POI搜索封装},label={lst:amap-poi}]
import httpx
from dataclasses import dataclass

@dataclass
class POIResult:
    id: str
    name: str
    type: str
    address: str
    location: str  # "lng,lat"
    rating: float
    cost: float  # average cost per person
    tel: str
    biz_ext: dict

class AmapClient:
    BASE = "https://restapi.amap.com/v3"

    def __init__(self, key: str):
        self.key = key
        self.http = httpx.AsyncClient(timeout=10.0)

    async def search_poi(
        self,
        keywords: str,
        city: str,
        location: str = "",
        radius: int = 5000,
        types: str = "",
        page: int = 1,
        page_size: int = 25,
    ) -> list[POIResult]:
        params = {
            "key": self.key,
            "keywords": keywords,
            "city": city,
            "offset": page_size,
            "page": page,
            "extensions": "all",
        }
        if location:
            params["location"] = location
            params["radius"] = radius
            params["sortrule"] = "distance"
        if types:
            params["types"] = types

        resp = await self.http.get(
            f"{self.BASE}/place/text", params=params
        )
        data = resp.json()
        if data["status"] != "1":
            raise AmapError(data.get("info", "Unknown"))

        results = []
        for poi in data.get("pois", []):
            biz = poi.get("biz_ext", {})
            results.append(POIResult(
                id=poi["id"],
                name=poi["name"],
                type=poi["type"],
                address=poi.get("address", ""),
                location=poi["location"],
                rating=float(biz.get("rating", 0)),
                cost=float(biz.get("cost", 0)),
                tel=poi.get("tel", ""),
                biz_ext=biz,
            ))
        return results
\end{lstlisting}

\subsection{路线规划集成}

路线规划 API 支持三种交通模式。WePlan 根据两点之间的距离自动选择推荐的交通方式：

\begin{itemize}[leftmargin=2em]
  \item 距离小于 1km：推荐步行
  \item 距离 1-5km：推荐公交或骑行
  \item 距离大于 5km：推荐驾车或公交
\end{itemize}

路线规划的响应数据包含了详细的路段信息、预估时间、距离和交通费用，这些数据会被 RouteAgent 进一步加工和整合。

\subsection{天气查询集成}

天气 API 的集成相对简单，但在 WePlan 中扮演着重要的角色。天气信息不仅用于方案的可行性评估，还用于生成穿衣建议和活动提醒。

WePlan 的天气查询策略是在每次规划开始时进行一次天气查询，并将结果缓存到短期记忆中。对于同一城市同一天的多次规划请求，系统直接使用缓存结果，避免重复 API 调用。

\subsection{API 配额管理}

高德地图开放平台对 API 调用有配额限制（免费额度为每日 5000 次）。WePlan 通过以下策略管理 API 配额：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{请求合并}：将多个小请求合并为一个大请求（如一次搜索多个类型的POI）
  \item \textbf{结果缓存}：对相同参数的请求结果缓存 1 小时
  \item \textbf{配额监控}：实时监控当日已用配额，当使用量超过 80\% 时触发告警
  \item \textbf{优雅降级}：当配额耗尽时，使用本地缓存的热门 POI 数据提供基础服务
\end{enumerate}

from fastapi import FastAPI, Request

app = FastAPI(title="WePlan API", version="1.0.0")

\subsection{编排器设计}

Orchestrator（编排器）是后端的核心组件，负责根据 Agent DAG 调度各 Agent 的执行顺序。编排器实现了以下关键功能：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{DAG 拓扑排序}：根据 Agent 之间的依赖关系确定执行顺序
  \item \textbf{并行调度}：没有依赖关系的 Agent 并行执行，最大化吞吐
  \item \textbf{状态管理}：跟踪每个 Agent 的执行状态（等待/执行中/完成/失败）
  \item \textbf{超时控制}：为每个 Agent 设置执行超时，防止单点阻塞
  \item \textbf{事件广播}：将 Agent 的执行状态实时广播给前端
\end{enumerate}

\subsection{性能优化}

后端的性能优化集中在以下几个方面：

\textbf{连接池管理}：对 MiniMax API、高德 API 和数据库连接使用连接池，避免频繁建立和断开连接的开销。

\textbf{异步并发}：利用 Python asyncio 实现高效的异步并发，单个 Worker 即可处理数百个并发请求。

\textbf{结果缓存}：对热门查询结果（如热门景点、常见路线）使用 Redis 缓存，缓存命中率约 35\%。

\textbf{响应压缩}：启用 gzip 压缩，减少传输数据量约 60\%。

\subsection{双面板设计}

WePlan 的前端采用独特的双面板布局，左侧为对话面板，右侧为方案展示面板。这一设计灵感来源于 Cursor IDE 的 Side Panel 和 Notion AI 的分栏模式。

\textbf{左侧对话面板}：用户通过自然语言与 AI 交互，支持多轮对话。面板实时显示 Agent 的思考过程（Agent Thinking），让用户了解 AI 的决策逻辑。

\textbf{右侧方案面板}：展示 AI 生成的规划方案，包含方案卡片、地图路线、时间轴、费用明细和雷达图评分。用户可以在方案之间切换对比。

双面板的核心交互逻辑是"对话驱动方案更新"：用户在左侧的每次输入都会触发右侧方案的增量更新，而右侧的方案操作（如点击POI查看详情）也会在左侧生成对应的对话气泡。

\subsection{雷达图评分}

每个规划方案都附带一个五维雷达图，直观展示方案在不同维度上的表现：

\begin{enumerate}[leftmargin=2em]

  \item \textbf{性价比}（Value）：费用与体验的比值
  \item \textbf{便捷性}（Convenience）：交通和路线的便利程度
  \item \textbf{舒适度}（Comfort）：整体行程的舒适和轻松程度
  \item \textbf{新鲜感}（Novelty）：活动的新颖性和独特性
\end{enumerate}

雷达图使用 Recharts 的 RadarChart 组件实现，支持鼠标悬浮查看具体分值和动态切换不同方案的对比视图。

\subsection{Agent Thinking 可视化}

Agent Thinking 可视化是 WePlan 前端最具特色的功能之一。在规划过程中，系统通过 SSE 实时推送每个 Agent 的思考过程，前端将这些信息以时间线的形式展示在对话面板中。

用户可以看到：
\begin{itemize}[leftmargin=2em]

这一设计不仅提升了用户等待过程中的体验，更重要的是建立了用户对 AI 决策过程的信任感。用户能够理解为什么 AI 推荐了某个方案，从而更愿意采纳推荐结果。

\section{数据模型}

\subsection{核心实体关系}

WePlan 的数据模型围绕以下核心实体构建：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{User}：用户信息和偏好配置
  \item \textbf{Session}：对话会话，包含上下文信息
  \item \textbf{Plan}：规划方案，一次请求可生成多个方案
  \item \textbf{Activity}：活动条目，归属于某个方案
  \item \textbf{POI}：兴趣点缓存，来自高德API
  \item \textbf{Vote}：投票记录，用于群体决策
\end{enumerate}

\subsection{JSONB 字段设计}

WePlan 大量使用 PostgreSQL 的 JSONB 类型存储半结构化数据。这一设计选择基于以下考虑：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{灵活性}：Agent 的输出格式可能随版本迭代而变化，JSONB 不需要 DDL 变更
  \item \textbf{查询能力}：PostgreSQL 的 JSONB 支持索引和丰富的查询操作符
  \item \textbf{性能}：JSONB 以二进制格式存储，读取性能优于纯文本 JSON
\end{enumerate}

\section{缓存策略}

\subsection{多级缓存架构}

WePlan 实现了三级缓存架构，以在数据新鲜度和访问性能之间取得平衡：

\textbf{L1 -- 进程内缓存}：使用 Python 的 lru\_cache 装饰器实现，容量 1000 条，TTL 5 分钟。主要缓存高频访问的配置数据和热门 POI 列表。

\textbf{L2 -- Redis 缓存}：使用 Redis 的 String 和 Hash 类型实现，TTL 1 小时。主要缓存 API 调用结果（POI 搜索结果、路线规划结果、天气数据）。

\textbf{L3 -- 数据库查询缓存}：利用 PostgreSQL 的查询计划缓存和 SQLAlchemy 的 Identity Map，减少重复查询。

\section{POI 数据增强}

\subsection{数据补全策略}

高德 API 返回的 POI 数据存在部分字段缺失的情况，WePlan 实现了多层次的数据补全策略：

\textbf{第一层：API 扩展查询。}对于缺失评分或人均消费的 POI，系统使用 POI ID 发起详情查询（detail API），获取更完整的商户信息。

\textbf{第二层：类型推断。}对于仍然缺失人均消费的 POI，系统基于 POI 类型和所在区域的统计数据进行推断。例如，朝阳区咖啡馆的平均人均消费约为 45 元。

\textbf{第三层：默认值填充。}作为最后的兜底策略，使用预设的安全默认值填充缺失字段。

\subsection{POI 质量评分}

WePlan 为每个 POI 构建了一个综合质量评分（Quality Score），综合考虑多个维度的数据质量：

\begin{equation}
Q_{poi} = 0.35 \cdot S_{rating} + 0.25 \cdot S_{reviews} + 0.20 \cdot S_{freshness} + 0.10 \cdot S_{completeness} + 0.10 \cdot S_{photos}
\end{equation}

其中 $S_{rating}$ 为归一化评分，$S_{reviews}$ 为评价数量得分（取对数归一化），$S_{freshness}$ 为数据新鲜度（基于最近评价时间），$S_{completeness}$ 为数据完整度（已填充字段比例），$S_{photos}$ 为图片丰富度。

质量评分低于 0.3 的 POI 将被过滤，不会出现在推荐结果中。

\section{数据安全}

\subsection{敏感数据处理}

WePlan 对用户数据的处理遵循最小权限原则：

\begin{itemize}[leftmargin=2em]
  \item 用户的精确位置信息仅在规划过程中使用，不做持久化存储
  \item 对话记录中的敏感信息（如手机号、身份证号）在入库前进行脱敏处理
  \item API 密钥通过环境变量注入，不在代码中硬编码
  \item 数据库连接使用 SSL 加密
\end{itemize}

\subsection{吞吐量分析}

系统在单机部署（4核 8GB）条件下的吞吐量瓶颈分析：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{LLM 推理}：每次请求需调用 MiniMax API 4-6 次，单次调用平均耗时 800ms，是主要瓶颈
  \item \textbf{高德 API}：每次请求调用 2-4 次，单次平均耗时 200ms
  \item \textbf{本地计算}：路线优化和方案生成的本地计算耗时约 50ms，可忽略
\end{enumerate}

通过 Agent 并行调度，系统将串行链路中可并行的部分（如 POIAgent 和 WeatherAgent）同时执行，整体耗时从理论串行的 8-10 秒优化到了 3-5 秒。

\section{商业模式}

\subsection{价值主张}

WePlan 的商业价值建立在以下核心价值主张之上：

\textbf{对用户}：将「周末不知道干什么」的焦虑转化为「选一个喜欢的方案出发」的轻松体验，节省平均 40 分钟的规划时间。

\textbf{对商户}：提供精准的曝光机会，让优质商户在用户决策的关键时刻被推荐。

\textbf{对平台（美团）}：构建从「规划」到「消费」的完整闭环，提升用户在美团平台上的停留时长和消费转化率。

\subsection{收入模型}

WePlan 的收入来源可以分为三个层次：

\textbf{层次一：流量导入分佣（短期）。}每个推荐方案中的 POI 如果链接到美团平台并产生消费，WePlan 可以获得一定比例的佣金分成。按平均每单佣金 5 元、月活用户 100 万、转化率 15\% 估算，月收入约 75 万元。

\textbf{层次二：商户推广服务（中期）。}为商户提供付费的优先推荐位，类似于美团现有的「推广通」产品。定价模式采用 CPC（每次点击付费），预估单次点击价格 2-5 元。

\textbf{层次三：增值服务（长期）。}提供会员制的高级功能，如无限次规划、独家优惠、专属客服等。月会员费预设为 9.9 元/月。

\section{安全设计}

\subsection{安全架构}

WePlan 的安全设计遵循纵深防御（Defense in Depth）原则，在每个层次都部署了相应的安全措施：

\textbf{网络层}：HTTPS 全链路加密，Nginx 配置 TLS 1.3，禁用不安全的密码套件。

\textbf{应用层}：输入校验、SQL 注入防护（通过 ORM 参数化查询）、XSS 防护（Content-Security-Policy 头）。

\textbf{数据层}：敏感数据加密存储、数据库访问控制、审计日志。

\textbf{运维层}：密钥轮换、漏洞扫描、安全监控告警。

\subsection{Prompt 注入防护}

作为一个 LLM 驱动的应用，Prompt 注入是 WePlan 面临的核心安全威胁。系统实现了多层防护机制：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{输入过滤}：使用正则表达式和关键词黑名单过滤明显的注入尝试
  \item \textbf{角色隔离}：System Prompt 和 User Prompt 严格分离，使用不同的消息角色
  \item \textbf{输出验证}：对 LLM 的输出进行结构化验证，拒绝不符合预期格式的响应
  \item \textbf{权限最小化}：LLM 只能调用预定义的工具函数，无法执行任意代码
\end{enumerate}

\section{隐私保护}

\subsection{隐私设计原则}

WePlan 的隐私保护遵循以下原则：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{数据最小化}：只收集实现功能所必需的最少数据
  \item \textbf{目的限定}：数据仅用于声明的用途，不做二次利用
  \item \textbf{用户控制}：用户可以查看、导出和删除自己的数据
  \item \textbf{透明告知}：在收集数据前明确告知用户数据的用途和存储方式
\end{enumerate}

\subsection{位置数据处理}

位置数据是 WePlan 最敏感的数据类型。系统对位置数据的处理策略如下：

\begin{itemize}[leftmargin=2em]
  \item 精确位置（经纬度）仅在规划过程中使用，结束后立即丢弃
  \item 持久化存储只保留城市级别的粗粒度位置信息
  \item 位置数据不会发送给除高德地图以外的任何第三方
  \item 用户可以选择手动输入位置而非使用GPS定位
\end{itemize}

\subsection{LLM 安全治理}

作为一个以 LLM 为核心的应用，WePlan 在模型安全方面实施了全面的治理措施：

\textbf{输入侧治理}：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{长度限制}：用户输入限制在 2000 字符以内，防止 token 注入攻击
  \item \textbf{敏感词过滤}：维护一份包含 500+ 敏感词的黑名单，过滤违规输入
  \item \textbf{注入检测}：使用基于规则的检测器识别常见的 Prompt 注入模式，如"忽略上面的指令"、"你现在是另一个AI"等
  \item \textbf{频率限制}：单用户每分钟最多 10 次请求，防止滥用
\end{enumerate}

\textbf{输出侧治理}：

\begin{enumerate}[leftmargin=2em]
  \item \textbf{格式校验}：所有 Agent 的输出必须通过 Pydantic 模型校验，拒绝不符合预期格式的响应
  \item \textbf{内容审核}：对生成的方案文本进行合规性审核，过滤可能的不当内容
  \item \textbf{幻觉检测}：通过与高德 API 返回的真实数据交叉验证，检测并过滤 LLM 幻觉（如虚构的POI名称）
  \item \textbf{置信度过滤}：对 Function Calling 的参数置信度进行评估，低置信度的调用触发确认流程
\end{enumerate}
